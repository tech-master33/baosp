# Contributing to BAOSP

Thank you for contributing to BAOSP — Blind Android Open Source Project.
This guide is written to work well with screen readers and keyboard-only navigation.
Every step is numbered and linear — no visual layout is assumed.

---

## Ways to contribute

You do not need to write code to contribute. Here are all the ways you can help:

1. Report a bug — describe something that does not work as expected
2. Request a feature — describe something that would help blind users
3. Improve documentation — fix unclear steps or add missing information
4. Test builds — install nightly APKs and report what you find
5. Translate — help make BAOSP accessible in more languages
6. Write code — fix bugs or add features in andrdscren, aotts, or the AOSP patches

---

## Before you start

You need:

1. A GitHub account — create one free at github.com/join
2. Git installed on your computer — download at git-scm.com

For code contributions to the AOSP build system, you also need:

3. A Linux computer or virtual machine (Ubuntu 20.04 or 22.04 recommended)
4. At least 400 GB of free disk space for the AOSP source tree
5. At least 16 GB of RAM
6. A fast internet connection — the AOSP source download is about 100 GB

For contributions to the apps only (andrdscren or aotts), see their own guides:

- Screen reader: github.com/tech-master33/andrdscren/blob/main/CONTRIBUTING.md
- TTS engine: github.com/tech-master33/aotts/blob/main/CONTRIBUTING.md

## Reporting a bug or requesting a feature

You do not need to know how to code to do this. It is one of the most valuable contributions.

1. Open github.com/tech-master33/baosp/issues
2. Activate New issue
3. Choose Bug report or Feature request
4. Fill in the title with one short sentence describing the problem or request
5. Fill in the body with as much detail as you can:

For a bug report include:
- What you were trying to do
- What happened instead of what you expected
- Whether it happens every time or only sometimes
- Your Android version and device model
- Which version of the BAOSP APKs you installed (check the nightly release tag)

For a feature request include:
- What task are you trying to accomplish
- How you currently work around not having this feature
- Who else would benefit from this feature

## Contributing documentation

Documentation changes do not require building Android or any special setup.

1. Fork the repository on GitHub (activate Fork near the top of github.com/tech-master33/baosp)
2. Clone your fork:

```bash
git clone https://github.com/YOUR-USERNAME/baosp.git
cd baosp
```

3. Create a branch:

```bash
git checkout -b docs/describe-your-change
```

4. Edit the Markdown file you want to improve using any text editor
5. Commit:

```bash
git add .
git commit -m "docs: clarify install steps for blind users on Android 8"
```

6. Push and open a pull request (see Step 7 in the code section below)

## Contributing code — AOSP patches

AOSP patches go into the `patches/` directory of this repository.
Each patch is a `.patch` file generated by `git format-patch`.

### Step 1 — Fork and clone

```bash
git clone https://github.com/YOUR-USERNAME/baosp.git
cd baosp
git remote add upstream https://github.com/tech-master33/baosp.git
```

### Step 2 — Create a branch

```bash
git checkout -b fix/describe-your-change
```

### Step 3 — Set up the AOSP source tree (if working on AOSP patches)

```bash
# Install repo tool
mkdir -p ~/bin
curl https://storage.googleapis.com/git-repo-downloads/repo > ~/bin/repo
chmod +x ~/bin/repo
export PATH=~/bin:$PATH

# Download AOSP source — this takes several hours
mkdir -p ~/aosp
cd ~/aosp
repo init -u https://android.googlesource.com/platform/manifest \
  -b android-14.0.0_r1 --depth=1
repo sync -j$(nproc) --no-clone-bundle

# Apply BAOSP patches
for patch in ~/baosp/patches/*.patch; do
  patch -p1 < "$patch"
done
```

### Step 4 — Make and test your change

Make your change in the AOSP tree.
Test it by building a minimal image:

```bash
source build/envsetup.sh
lunch aosp_x86_64-userdebug
make -j$(nproc) systemimage
```

### Step 5 — Generate a patch file

```bash
# In the AOSP directory, after committing your change in the affected module:
git format-patch HEAD~1 -o ~/baosp/patches/
```

Then commit the new `.patch` file to your baosp branch.

### Step 6 — Write a clear commit message

```bash
git add patches/
git commit -m "fix: enable accessibility service on first boot for API 33+

On Android 13 and above the SettingsProvider overlay key changed.
Updated baosp_accessibility_defaults.xml to use the new key name."
```

### Step 7 — Push and open a pull request

```bash
git push origin your-branch-name
```

Then:

1. Open github.com/YOUR-USERNAME/baosp
2. Activate Compare and pull request
3. Title: one sentence describing the change
4. Description: what problem does this solve, what did you test, any known limitations
5. Activate Create pull request

## Accessibility rules for all code contributions

Every code change to this project must follow these rules.
These are not optional — they are the purpose of the project.

1. All UI elements must have a content description so the screen reader announces them correctly
2. All interactive elements must be reachable without a touchscreen — keyboard and switch access must work
3. Touch targets must be at least 48dp wide and 48dp tall so they are easy to activate
4. Do not use color alone to show information — also use text, shape, or position
5. Focus order must be logical — navigating by swipe or keyboard must follow a natural reading order
6. All strings shown to the user must be in the `strings.xml` resource file so they can be translated
7. Test with the screen reader turned on before submitting — if the screen reader cannot use your change, it will not be accepted

## Code review process

After you open a pull request:

1. A maintainer will read your changes and may ask questions in the comments
2. Reply to comments — activate the Resolve conversation button once you have addressed the point
3. If changes are requested, push new commits to the same branch — the pull request updates automatically
4. Once approved, a maintainer will merge your pull request

Most pull requests receive a first response within a few days.
If you have not heard back after a week, add a comment to the pull request to ask for an update.
